# 校园 AI 学长 · 高校社区里的智能问答与找帖

做这个项目的起因挺具体的：大学里的信息太散了——选课、宿舍、四六级、二手、组队，答案散在各种群聊和往届学长的脑子里，想搜也搜不到。我想做的东西很明确：一个能用大白话问、答案还得靠谱、并且能从站内真实帖子里帮你把东西捞出来的"AI 学长"。

最后它长成了一个完整的高校匿名社区小程序，AI 学长只是其中我啃得最狠的一块；外面还包着帖子社区、实时私信、内容审核、运营后台和一整套监控告警。这是我和一个同学两个人做的，我是全栈主力，后端和那个 AI 服务基本是我一个人写的，差不多做了一个月。

这个仓库放的不是全部代码，而是我自己觉得值得拿出来讲的几处设计和踩过的坑，配了些从真实源码里摘出来、抹掉密钥和内网地址的片段。完整的业务代码在团队的私有仓库里。

## 技术上我自己最来劲的几块

懒得堆功能清单，就讲我花时间最多、面试也最想聊的几处：

- **让 AI 找帖不"瞎答"。** 最早它会把内容只有"测试"俩字的帖子也当成匹配返回。我后来发现这不是阈值能解决的，根子在向量检索本身，于是换了个提问方式加了一道判定，才真正治住。展开在 [docs/ai-service.md](./docs/ai-service.md)。
- **把内网那台单槽大模型伺候明白。** 对话模型跑在内网一台机器上、同时只能处理一个请求，特别容易被打爆。我围着这个瓶颈做了一串取舍：关掉 SDK 自动重试、给整轮加超时预算并优雅降级、砍掉没必要的模型调用。
- **后台权限跟学生端彻底隔开。** 管理员是另一套账号、另一套 JWT 密钥，学生的 token 拿不到任何后台接口。这块在 [docs/backend.md](./docs/backend.md)。
- **数据进知识库前先脱敏。** 数据源里全是学生隐私，管道默认只写本地文件不写库，人工抽检确认了才推送，脱敏放在最前面强制做。

## 系统大致长这样

```
   [ 微信小程序 ]            [ 运营后台 · Vue ]
         \                       /
          Nginx 同源反代  (/ · /api · /grafana)
                      |
        +-------------+--------------+
        |                            |
  Spring Boot 后端              AI 微服务 (FastAPI + LangGraph)
  用户 / 帖子 / 私信            知识问答 · 找帖 · AI 发帖 · 异步记忆
  审核 / 管理端 RBAC                  |
        |                      Neo4j 知识图谱 + 双向量索引
        |                            ^
        |                   数据管道（清洗 + 脱敏后入库）
        v
  MySQL · Redis · MinIO              业务数据 / 缓存 / 对象存储
        |
  Prometheus + Grafana + Alertmanager  →  企微 / 钉钉告警
```

后端是个 Spring Boot 单体，按领域分包；AI 那块单独拆成 Python 服务，两边走 HTTP 解耦，这样它能单独跑在带 GPU 的机器上。整套中间件和监控是一份 Docker Compose 起的。更细的拆分和数据流见 [docs/architecture.md](./docs/architecture.md)。

## 大概多大规模

后端是 Spring Boot 单体，285 个 Java 文件、约 2.4 万行，33 个 Controller、75 个 Service、约 38 张表、20 个数据库迁移脚本。AI 服务是 Python，约 5800 行、对外 18 个 HTTP 接口，配了 19 个测试文件 98 个用例。小程序约 1.6 万行、32 个页面；运营后台约 2800 行、14 个权限页面；数据管道约 660 行。运维这边一份 compose 拉起 17 个容器。六个服务用 Git Submodule 聚合，加起来五万行上下。

AI 那块有几个我比较在意的数字：找帖经过判定之后的 precision 到 0.97，"绝对不该出现"的帖子漏出是 0；知识问答的对抗样本拒答率 100%；意图识别调完提示词从 95% 提到 100%。这些都是我自己标的评测金标集跑出来的。

## 关于这个仓

两个人的团队项目，我是全栈主力、主导了架构和大部分模块，但不是一个人全包，具体分工在[那篇贡献说明](./docs/portfolio.md)里写了。文里的数字我尽量标了来源：评测跑出来的实测值、还是我自己观察估计的经验值。项目是完整能跑、也用真实数据导过的，但用户量是校内试用级别——我做的优化都是冲着"内网单槽大模型"这个真实瓶颈去的，不是大流量分布式那一套，这点我不往大了说。代码片段都是从真实源码里摘出来、删掉密钥和无关细节的，留着能说明思路的部分。
